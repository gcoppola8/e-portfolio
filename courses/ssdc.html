My discussion in Unit 1

CSRF - Cross-site Request Forgery is an attack consisting of forging a request to a website where the user has logged in from the same browser, that through malicious code like a hidden form sends a request from the user's browser. The request could post a message on a forum, modify settings to introduce some vulnerability, send a payment and more. In OWASP Top 10 list CSRF was until 2017 a flaw by itself, but now is referred as the more generic "Injection" that include also other kind of vulnerabilities as SQL Injection and others.

In this activity diagram, the user is able to perform a normal action on a generic web page with a form. But the same scenario as shown is vulnerable to CSRF attacks as described above, an attacker can fill the form and send it from a third-party page as well escalating the session the user has.

The solution to CSRF attacks is to generate the form from the server with a token field pre set with a generated token, and at the moment the server receive and process the form by the user can verify the token validity.

In case an attacker tries the same attack, the server is going to reject the form submitted since it has no valid token into it.

Here an improved activity diagram: (in red the improved security measures)

Nowadays is possible to implement such security measure almost effortlessly since all the popular web framework of the major languages offer such feature out of the box or with minimal effort.

Why I believe an activity diagram can better represent such vulnerability and pretty much all the OWASP vulnerabilities, is because activity diagram is a behavioral UML model that in fact shows a dynamic aspect of the system, making possible to spot for determined actions a vulnerability. A structural model on the contrary wouldn't show an improper behavior. And structure could have no issues, but still have vulnerabilities.

Formative activity

Regex

Regular expressions are expressions to find or replace text using string and patterns. Stephen Cole Kleene (Kleene, 1952) designed regular expressions in the 50s, and nowadays, they are implemented in most programming languages.

Evil regular expressions are regular expressions whose patterns rely on chunks of the regular expression itself. It means that computational time working on these can rise exponentially. ReDOS attacks (Van Der Merwe et al., 2017) rely on this principle to occupy system resources with very long strings. As a result, the system becomes unavailable to the users as in other Denial Of Service attacks.

We can mitigate ReDos attacks in different ways by using regex analysis tools to identify evil regex and avoid utilising them or limiting the time spent by the system to compute a single regex.

My discussion in Unit 8

TrueCrypt was a popular open-source tool to encrypt and decrypt volumes, folders, and entire systems. From 2014 its author discontinued the maintenance of the tool, raising the risk of security issues for everybody relying on it. 
The source code of the last version released that is freely available has been ispected and vulnerabilities of different risk levels are found in the code (Junestam, A., Guigo, N., 2014). Making the product unusable nowadays, confirming the declaration of the author. 

The most criticals appear to be: 537 - Truecrypt 7 Derived Code/Windows: Incorrect Impersonation Token Handling EoP - project-zero (chromium.org)
and 
538 - Truecrypt 7 Derived Code/Windows: Drive Letter Symbolic Link Creation EoP - project-zero (chromium.org)

And for these reason I wouldn't suggest any friend to use TrueCrypt, but to evaluate some of the alternatives as Veracrypt or boxcryptor.
